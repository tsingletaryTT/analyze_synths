The story

From a library management problem to Blackhole silicon

The problem

A growing library of synthesizer pieces — long-form ambient tracks, generative drones, evolving textures — with no good way to understand what was in it. Standard tools give BPM and key. Not useful when the track has no tempo and the key is ambiguous.

The goal was a creative vocabulary: spacey, crystalline, droning, organic, oozy. Descriptors that mean something to a synthesizer composer.

v1 — Feature extraction

A librosa-based pipeline extracting 80+ features per track mapped through empirically-tuned thresholds to 176 descriptors: 9 core moods, 8 extended, 100 advanced, 59 character tags. K-means clustering, phase detection, and sequence recommendation on top.

It worked. The results felt right. The problem was speed on long files.

The insight

STFT is just three matrix multiplications. Given pre-computed static bases:

cos_proj = frame @ cos_basis
sin_proj = frame @ sin_basis
mag = sqrt(cos² + sin² + ε)
mel = mag @ mel_filter

Float32, precomputed matrices, batch frames — exactly what Tensix cores are for.

TT-Lang kernel

First target: TT-Lang simulator — a Python DSL for Tensix dataflow kernels. Two @ttl.operation kernels (a DataflowBuffer can’t be simultaneously written and read in one pass). All tile dimensions must be multiples of 32; n_freqs=1025 padded to 1056.

Loop ordering between reader and compute threads must be identical or tiles arrive out of sequence and produce silently wrong results.

Hardware dispatch

Four P300C Blackhole chips in a P150X4 mesh. Getting JAX to see them meant finding the right PJRT plugin binary. The tt-metal and tt-xla-venv paths both failed — wrong chip count, undefined symbols. The working path: pjrt_plugin_tt.so from tt-xla with the p300c-xla-test site-packages tree.

@jax.jit compiles the four-matmul STFT graph once and reuses it for every chunk.

Temporal narrative

Fast STFT opened a door. With 60 min processed in under a second, richer temporal analysis became practical: tension trajectories in 2-second windows, change-point section detection (7 section types), prose generation, and a cross-piece similarity graph linking structurally related tracks.

What’s next

The STFT bottleneck moved. Now everything else shows up.

Multi-chip parallelism

The P150X4 mesh has 4 Blackhole chips. Explicit device placement could run 4 chunks simultaneously — another 4x on top of per-chip speed, pushing STFT for 60 min of audio under 0.25 s.

Trajectory on hardware

TrajectoryAnalyzer still runs on CPU. Per-window tension computations share the same matmul structure as STFT — a JIT-compiled trajectory kernel would make the full pipeline hardware-accelerated end to end.

Real-time streaming

The chunk-streaming architecture was designed for this. With hardware STFT and fast trajectory, latency per 30-second chunk could drop below 10 ms — enabling live analysis as audio is being performed or recorded.

Grow the pool

The library has 8 pieces right now. With more music the similarity graph becomes genuinely informative, clustering finds real structure, and descriptor thresholds could be refined against a larger annotated set.

Similarity-driven sequencing

The CrossPieceSimilarity graph gives structural neighbors. Feeding it into the SequenceRecommender so transitions favor structurally complementary pieces rather than only spectrally similar ones would produce more interesting playlists.

Online learning

Mood descriptors and section thresholds are hand-tuned. A track could carry a manual annotation that updates the classifier’s priors. Over 100+ annotated pieces, thresholds would reflect the actual composer’s perception.
